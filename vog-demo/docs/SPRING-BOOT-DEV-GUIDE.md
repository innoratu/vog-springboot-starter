# Spring Boot Developer Guide (using `vog-demo`)

A hands-on beginner guide to **creating and evolving a Spring Boot project in VS
Code**. It uses this app (`vog-demo`) as the running example, so every concept
maps to real files you can open.

By the end you'll know:
- how to create a project with the **Spring Boot Extension Pack** in VS Code,
- **where every Spring Initializr choice ends up** in the generated files,
- how to **add/remove libraries (starters)** later,
- how to **change the baseline** (Java version, Spring Boot version, group/artifact/package, jar vs war),
- how to **run and debug** with the Spring Boot Dashboard.

> Official reference: <https://code.visualstudio.com/docs/java/java-spring-boot>

---

## 1. What is Spring Initializr?

**Spring Initializr** (<https://start.spring.io>) is a project generator. You give
it some metadata (group, artifact, Java version…) and pick "starters" (bundled
libraries), and it produces a ready-to-build skeleton: a `pom.xml`, a main class, a
config file, the Maven wrapper, etc.

You can use it three ways — they all produce the *same* files:
1. The website <https://start.spring.io>.
2. **Inside VS Code** via the Spring Boot Extension Pack (this guide's focus).
3. IntelliJ's built-in generator.

`vog-demo` itself was generated by Initializr (see the leftover `HELP.md` and
`.vscode/NEWLY_CREATED_BY_SPRING_INITIALIZR` marker file).

## 2. One-time VS Code setup

1. Install the **Extension Pack for Java** (Microsoft) — gives you Java language
   support, Maven, debugger.
2. Install the **Spring Boot Extension Pack** (VMware/Broadcom). It bundles:
   - **Spring Initializr Java Support** — create projects + add starters.
   - **Spring Boot Tools** — smart editing of `application.properties`, YAML, `@`-annotations.
   - **Spring Boot Dashboard** — run/stop/debug apps from a side panel.
3. Make sure a **JDK 17+** is available (this repo uses Java 17 via SDKMAN — see
   [`ENVIRONMENT.md`](ENVIRONMENT.md)).

## 3. Create a new project in VS Code (step by step)

> This is how `vog-demo` was born. You don't need to redo it — follow along to
> understand each prompt.

1. Open the Command Palette: **`Ctrl+Shift+P`**.
2. Run **`Spring Initializr: Create a Maven Project`**.
   (There's also a Gradle variant; we use Maven.)
3. Answer the prompts, in order:

| Prompt | What it means | `vog-demo`'s answer |
|--------|---------------|---------------------|
| **Spring Boot version** | Framework version | `4.1.0` |
| **Language** | JVM language | `Java` |
| **Group Id** | Your org's reverse-domain namespace | `com.vog.example` |
| **Artifact Id** | The project/module name | `vog-demo` |
| **Packaging type** | `Jar` (standalone app) or `War` (deploy to a server) | `Jar` |
| **Java version** | JDK to compile/run against | `17` |
| **Dependencies** | Starters to include (search & multi-select) | Web, JPA, H2, Validation… |

4. Choose a folder to generate into, then click **"Open"** when prompted.

That's it — VS Code writes the skeleton and opens it.

## 4. Where does each Initializr choice go? (the important part)

Every answer above is written into a specific place. Open `vog-demo/pom.xml`
alongside this table:

| Initializr field | Lands in | In `vog-demo` |
|------------------|----------|---------------|
| Spring Boot version | `<parent>` → `<version>` in `pom.xml` | `4.1.0` |
| Group Id | `<groupId>` | `com.vog.example` |
| Artifact Id | `<artifactId>` **and** the root folder name | `vog-demo` |
| Name / Description | `<name>` / `<description>` | `vog-demo` |
| Packaging | `<packaging>` (absent = `jar` default) | jar |
| Java version | `<properties>` → `<java.version>` | `17` |
| Dependencies (starters) | `<dependencies>` block (one `<dependency>` each) | web, data-jpa, validation, h2, … |
| Base package | Folder path under `src/main/java` + `package` in the main class | `com.vog.example.vog_demo` |

And these files are generated regardless of your choices:

```
vog-demo/
├── pom.xml                     # ← all metadata + dependencies + build config
├── mvnw, mvnw.cmd              # Maven Wrapper (runs the right Maven version)
├── .mvn/wrapper/...            # which Maven version the wrapper downloads
├── .gitignore                  # sensible Java/Maven ignores
├── HELP.md                     # generated help (safe to delete)
└── src/
    ├── main/java/<package>/VogDemoApplication.java   # entry point (@SpringBootApplication)
    ├── main/resources/application.properties          # runtime configuration
    ├── main/resources/static/  templates/             # (empty) web assets
    └── test/java/<package>/VogDemoApplicationTests.java
```

> **Key takeaway:** there is no hidden database of settings. Everything Initializr
> asked you is plain text in **`pom.xml`** (plus the folder/package layout). To
> change anything later, you edit `pom.xml` — see §6–7.

## 5. The generated files, explained

- **`pom.xml`** — the Maven "Project Object Model". Declares the parent, your
  coordinates, Java version, dependencies, and build plugins. This is the file you
  edit most for project-level changes.
- **`VogDemoApplication.java`** — the entry point. The `@SpringBootApplication`
  annotation switches on auto-configuration and component scanning; `main()` boots
  the embedded server.
- **`application.properties`** — key/value runtime config (database URL, ports,
  logging, etc.). YAML (`application.yml`) is an equivalent alternative.
- **`mvnw` / `mvnw.cmd`** — the **Maven Wrapper**: run `./mvnw ...` instead of a
  system `mvn` so everyone uses the same Maven version (defined in
  `.mvn/wrapper/maven-wrapper.properties`).
- **`HELP.md`** and **`.vscode/NEWLY_CREATED_BY_SPRING_INITIALIZR`** — generated
  hints/markers; delete whenever you like.

## 6. Understanding `pom.xml` (the control panel)

Open `vog-demo/pom.xml`. The parts you'll touch:

```xml
<parent>                                  <!-- inherits sensible defaults + versions -->
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-parent</artifactId>
  <version>4.1.0</version>                <!-- ← Spring Boot version -->
</parent>

<groupId>com.vog.example</groupId>        <!-- ← your coordinates -->
<artifactId>vog-demo</artifactId>
<version>0.0.1-SNAPSHOT</version>

<properties>
  <java.version>17</java.version>         <!-- ← Java version -->
</properties>

<dependencies> ... </dependencies>        <!-- ← libraries (starters), see §7 -->

<build>
  <plugins>
    <plugin>                              <!-- packages a runnable jar, enables spring-boot:run -->
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-maven-plugin</artifactId>
    </plugin>
  </plugins>
</build>
```

Two ideas that make this less scary:
- **The parent manages versions for you.** Notice most `<dependency>` entries have
  **no `<version>`** — the `spring-boot-starter-parent` picks compatible versions.
  You only specify a version for libraries the parent doesn't manage (in this repo,
  `springdoc-openapi` has an explicit `<version>` for that reason).
- **A "starter" is a curated bundle.** `spring-boot-starter-web` pulls in Spring
  MVC, Jackson (JSON), and an embedded Tomcat — one line, many libraries.

**After editing `pom.xml`, reload:** Command Palette → **`Java: Reload Projects`**
(or click the Maven refresh). This re-reads dependencies.

## 7. Adding and removing libraries (starters)

### Option A — VS Code "Add Starters" (recommended for beginners)
1. Open `pom.xml`.
2. Command Palette → **`Spring Initializr: Add Starters`**
   (or right-click in `pom.xml` → *Add Starters…*).
3. Search and tick the starters you want; confirm.
4. VS Code shows a **diff** of the `pom.xml` changes — accept it.
5. Run **`Java: Reload Projects`** so the new libraries download.

### Option B — edit `pom.xml` by hand
Add a `<dependency>` inside `<dependencies>`. Example — adding Spring Security:
```xml
<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-security</artifactId>
</dependency>
```
Then **`Java: Reload Projects`**. Remove a library by deleting its `<dependency>`
block and reloading.

### Verify from the terminal
```bash
./mvnw dependency:resolve        # downloads/validates dependencies
./mvnw dependency:tree           # shows the full dependency graph
```

### Common starters you'll reach for

| Starter | Adds |
|---------|------|
| `spring-boot-starter-web` | REST + MVC + embedded Tomcat (servlet) |
| `spring-boot-starter-webflux` | Reactive web (non-servlet) |
| `spring-boot-starter-data-jpa` | JPA/Hibernate ORM + repositories |
| `spring-boot-starter-validation` | Bean Validation (`@NotBlank`, …) |
| `spring-boot-starter-security` | Authentication/authorization |
| `spring-boot-starter-actuator` | Health checks, metrics endpoints |
| `spring-boot-starter-test` | JUnit 5, Mockito, AssertJ, Spring test |
| DB drivers | `com.h2database:h2`, `org.postgresql:postgresql`, … |

## 8. Changing the baseline

### Change the Java version
1. Edit `<java.version>` in `pom.xml` (e.g. `17` → `21`).
2. Make sure that JDK is installed and active (`sdk install java 21.0.11-tem`,
   then `sdk use ...`). See [`ENVIRONMENT.md`](ENVIRONMENT.md).
3. **`Java: Reload Projects`**, then `./mvnw clean compile`.

### Change the Spring Boot version
1. Edit the `<version>` inside `<parent>` (e.g. `4.1.0`).
2. Reload + rebuild. **Watch for breaking changes** across major versions — e.g.
   Boot 4 moved some packages (see this repo's memory note about test-slice
   packages). Read the release notes when bumping a major version.

### Change coordinates or package
- **Group/Artifact/Name/Description:** edit the matching `pom.xml` elements. Renaming
  the artifact usually means renaming the root folder too.
- **Base package:** rename the folders under `src/main/java` (VS Code's *Rename*
  refactor updates the `package` statements and imports).

### Change packaging (jar ↔ war)
- Add/adjust `<packaging>war</packaging>` if you must deploy to an external servlet
  container. For most modern apps, keep the default **jar** (self-contained,
  `java -jar target/vog-demo-0.0.1-SNAPSHOT.jar`).

### Swap the embedded web server (Tomcat → Jetty / Undertow)
Spring Boot runs an **embedded web server** inside the app. `spring-boot-starter-web`
doesn't hard-wire it — it just *transitively* pulls in `spring-boot-starter-tomcat`,
and Boot auto-configures whichever servlet container it finds on the classpath. So
you can swap servers with a **`pom.xml` change and no code changes** in a typical
app.

Example — switch from Tomcat (default) to **Undertow**:
```xml
<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-web</artifactId>
  <exclusions>
    <exclusion>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-tomcat</artifactId>
    </exclusion>
  </exclusions>
</dependency>
<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-undertow</artifactId>
</dependency>
```
Reload the project and rebuild — the same `/api/...` endpoints now run on Undertow.
For **Jetty**, use `spring-boot-starter-jetty` instead of the Undertow starter.

Options and when to pick them:

| Server | Notes |
|--------|-------|
| **Tomcat** (default) | Most battle-tested, best docs — keep unless you have a reason to change |
| **Undertow** | Lightweight, low memory, high throughput — popular for microservices |
| **Jetty** | Flexible embedding, strong WebSocket support |

Things to watch (the small non-free part):
- **Container-specific config doesn't carry over:** `server.tomcat.*` properties
  become `server.jetty.*` / `server.undertow.*`. Generic ones (`server.port`,
  `server.compression.*`) work on all of them.
- **Container-specific code needs porting:** Tomcat `Valve`s, a
  `TomcatServletWebServerFactory` customizer, or JSP (Undertow has no JSP). This app
  uses none of those, so it would be a clean swap.
- **Reactive apps differ:** a WebFlux (reactive) app defaults to **Netty** and swaps
  among the reactive server variants instead. `vog-demo` is servlet MVC, so the
  Tomcat/Jetty/Undertow trio applies here.

For most apps the performance difference is marginal — stay on Tomcat unless a
specific requirement pushes you elsewhere.

## 9. Running and debugging in VS Code

- **Spring Boot Dashboard** (left activity bar): lists your apps. Click ▶ to run,
  ■ to stop, or the bug icon to debug. It also shows live **beans** and **request
  mappings** once running.
- **Terminal equivalent:** `./mvnw spring-boot:run`.
- **Package a runnable jar:** `./mvnw clean package` → `target/*.jar` →
  `java -jar target/vog-demo-0.0.1-SNAPSHOT.jar`.
- Edit `application.properties` with Spring Boot Tools active — you get
  autocompletion and docs for every property.

## 10. How this all shows up in `vog-demo` (worked examples)

This project already demonstrates the operations above:

- **Started from Initializr** with `com.vog.example` / `vog-demo` / Java 17 / Boot 4.1.0.
- **Baseline was adjusted:** we removed the starters that didn't fit
  (`webflux`, `data-rest`, `restclient`) and added the ones we needed
  (`web`, `data-jpa`, `validation`, `h2`, `springdoc-openapi`) — exactly the §7
  workflow. See `pom.xml`.
- **An explicit dependency version** was needed for `springdoc-openapi` because the
  parent doesn't manage it — an example of the §6 "version management" rule.
- **Test starters were added** for Boot 4's per-technology test slices
  (`spring-boot-starter-webmvc-test`, `spring-boot-starter-data-jpa-test`).
- **Configuration** lives in `src/main/resources/application.properties` (H2
  datasource, JPA, H2 console, Swagger path).

Open `pom.xml` and match each `<dependency>` to the tables in §7 — that's the whole
picture.

## 11. Try it yourself (exercises)

Small, safe changes to practice the workflow. After each: reload, run
`./mvnw test`, and start the app.

1. **Add Actuator.** Add `spring-boot-starter-actuator`, then open
   `http://localhost:8080/actuator/health`. (Expect `{"status":"UP"}`.)
2. **Add a config property.** Put `server.port=8081` in `application.properties`,
   restart, confirm the app now serves on 8081.
3. **Add a field.** Give `Organism` a `conservationStatus` field (entity + DTO +
   seeder), reload, and see it in `GET /api/organisms`.
4. **Bump Java to 21.** Install JDK 21 via SDKMAN, change `<java.version>`, update
   `.sdkmanrc`, rebuild.

Each of these touches exactly one layer — a good way to learn where things live.
